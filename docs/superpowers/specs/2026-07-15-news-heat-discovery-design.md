# 新聞熱度發現層（主流族群／焦點股）設計

日期：2026-07-15
狀態：已與使用者確認方案，待實作

## 目的

新聞熱度作為**發現層（discovery）**：從近期新聞中分析出主流族群與焦點股票，
供使用者在這些股票回檔修正時尋找進場點，避免追高。

明確的非目標：

- **不進評分**。名稱匹配結果不寫 `news_stock_map`，不流入
  `NewsRelatedRule` 與 `BatchDataLoader`——由架構保證，非靠約定。
- 不做即時盤中監控（App 定位為盤後掃描）。
- 不做情緒分析（熱度＝提及頻率，與標題情緒字眼無關）。

## 背景與語料證據（2026-07-15，production DB 6,198 筆標題）

- 既有「新聞→個股」關聯只認括號代碼格式，涵蓋的幾乎都是營收速報／
  Factset 等例行數據稿；炒作類新聞（「華新科、南亞漲逾30%」）用公司名
  不帶代碼，全數漏接。
- 公司簡稱匹配的風險分層（對全庫標題實測）：
  - 3 字以上名稱：~1,190 家，幾乎零歧義（台積電 1,153 次、聯發科 133 次）。
    需最長優先（長榮航≠長榮、聯發科≠聯發、統一超≠統一）。
  - 2 字名稱：1,496 家，高頻者是真焦點股（鴻海 145、聯電 61、台塑 56）
    但誤配嚴重：「大成」撞「最**大成**長」、「上海」是城市、「三星」是
    韓國三星、「世界」是普通名詞。
- 官方產業別（`stock_master.industry`）過粗：台積電與鴻海同屬
  「電子工業」，無法呈現「記憶體」「玻璃基板」等題材輪動。
- 新聞保留 30 天即清理：提及數歷史事後不可回補，這是快照表存在的理由。

## 架構總覽（方案 C：顯示即時算＋每日快照）

```
新聞標題（近 28 天，DB 即時讀）
   │
   ├─ StockNameMatcher（純函數）──→ per-symbol 提及數
   ├─ ThemeMatcher（純函數）─────→ per-題材 文章數 + 題材↔個股連結
   │
   ├─ HeatCalculator（純函數）───→ 焦點股排行（含爆量徽章）
   │                                主流族群排行
   │                                × 三模式狀態交叉（回檔標注）
   │
   ├─ 顯示：新聞頁「熱度分析」Tab（即時計算，不讀快照表）
   │
   └─ 每日更新尾端：news_mention_daily 快照 upsert（fail-safe，
      只為累積歷史供未來回測，顯示層不依賴）
```

- 顯示永遠用「當下字典」即時重算，口徑自我一致；
- 快照表獨立累積，30 天新聞清理不影響它；
- 快照寫入失敗只記 log，不中斷更新流程；
- 若未來熱度分頁棄用，刪表即可降級為純即時（不對稱風險：A→C 需資料
  回補、不可行；C→A 只是刪表）。

## 元件設計

### 1. StockNameMatcher（`lib/domain/services/news/stock_name_matcher.dart`）

- 字典建構：`stock_master` 現役股票中
  - 名稱長度 ≥ 3：全部納入；
  - 名稱長度 = 2：僅納入人工白名單（`NewsHeatParams.twoCharNameWhitelist`，
    實作時從語料高頻榜逐一人工審核，預估 30–50 個；語料重放稽核通過
    才定稿）。
- 匹配演算法：候選名稱按長度降冪排序，逐一在標題中找出現位置；
  已被較長名稱佔用的字元位置不得重複匹配（位置消耗）。同一標題同一
  股票多次出現計 1（去重為 per-article 提及）。
- `-KY` 等後綴名稱照 `stock_master.name` 原樣匹配。
- 輸出：`Map<String symbol, Set<String newsId>>`（或等價 per-article 結構）。

### 2. 題材字典與 ThemeMatcher（`lib/core/constants/news_heat_params.dart`）

- `Map<String theme, List<String> keywords>`，約 40 題材，各帶同義詞
  （例：記憶體＝[記憶體, DRAM, NAND, HBM]；AI＝[AI, 人工智慧]）。
  英文詞不分大小寫。
- 題材↔個股連結為資料導出：同一篇標題同時命中題材 T 與股票 S，
  即建立 (T, S) 連結；族群卡片的成分股＝該題材文章中提及次數 Top N。
  不維護手動的題材→成分股對照表。
- 維護方式：新題材出現時手動加詞（低頻）；字典異動時 bump
  `NewsHeatParams.dictionaryVersion`。

### 3. HeatCalculator（純函數）

- 窗口：近 7 天 vs 前 21 天（皆在 30 天保留窗內，無需等待歷史累積）。
- 焦點股：`mentions7d` 排序；爆量徽章條件＝近 7 天篇數達前 21 天
  **週均的 3 倍**（數學上等價於 `mentions7d >= mentionsPrev21d`），
  且 `mentions7d >= 3`（防低基數假爆量）。
- 主流族群：題材文章數，同窗口對比，同爆量邏輯。
- 三模式交叉：焦點股列表逐檔標注現行三模式（起漲／強勢／回檔）狀態
  徽章，資料源重用既有模式判定（與今日頁同源）；「回檔中」列突顯，
  並提供「只看回檔中」過濾——此為本功能的核心使用情境（回檔進場
  觀察清單）。
- 所有門檻集中於 `NewsHeatParams`，禁魔術數字。

### 4. 快照表 `news_mention_daily`

| 欄位 | 型別 | 說明 |
|------|------|------|
| date | DateTime | 本地日（新聞 publishedAt 的 local 日曆日） |
| kind | TEXT | `stock` 或 `theme` |
| key  | TEXT | symbol 或題材名 |
| mentionCount | INTEGER | 該日提及篇數 |
| dictionaryVersion | INTEGER | 寫入當下的字典版本 |

- 主鍵 (date, kind, key)。
- 寫入時機：每日更新尾端，**回補最近 3 個本地日**（重算後 upsert，
  晚到新聞自我修正、天然冪等）。
- fail-safe：寫入例外只記 log，更新流程照常完成。
- 顯示層不讀此表；唯一消費者是未來的回測（屆時按
  dictionaryVersion 取同版本區段，避免字典演化造成的假爆量）。

### 5. UI：新聞頁「熱度分析」Tab

- 新聞頁頂部分頁：「全部新聞｜熱度分析」。
- 熱度分析內容：
  1. **主流族群**：Top ~8 題材卡片（題材名、7 天篇數、對比前期的
     趨勢箭頭、爆量徽章、成分股 chips 點擊進個股詳情）。
  2. **焦點股**：Top ~20 列表（代碼＋名稱、提及數、🔥爆量徽章、
     三模式徽章；「回檔中」列突顯；「只看回檔中」切換；點擊進
     個股詳情）。
- 計算時機：切入 Tab 時即時計算（近 28 天標題 × 字典，估 <500ms；
  實測若超過則移入 isolate）。與新聞頁「重新整理」（先抓 RSS 再
  重讀）共用資料來源，刷新後熱度同步反映。
- 空狀態／載入狀態沿用既有 pattern。

## 錯誤處理

- 快照寫入：catch + `AppLogger.warning`，不 rethrow（同 RuleAccuracyService
  的 fail-safe 慣例）。
- 熱度計算異常：Tab 顯示錯誤狀態＋重試，不影響「全部新聞」分頁。
- 三模式資料缺失（例如尚未跑過分析）：焦點股列表正常顯示、僅省略
  模式徽章。

## 測試策略

1. **StockNameMatcher（TDD）**：最長優先（長榮航／長榮、聯發科／聯發、
   統一超／統一）、位置消耗不重複計分、2 字僅白名單、`-KY` 名稱、
   同篇多次出現計 1。
2. **語料重放稽核（白名單定稿閘）**：matcher 對 production 全庫標題重放，
   輸出 per-name 命中樣本人工抽驗；2 字白名單逐名審核後才定稿。
   誤配率量化記錄於實作 PR。
3. **HeatCalculator**：fixture 測試窗口切分、爆量門檻邊界（恰 3 篇、
   低基數）、排序穩定性。
4. **快照 DAO**：真 DB（in-memory drift）測試 upsert 冪等、3 日回補、
   (date,kind,key) 唯一性。
5. **UpdateService 整合**：快照寫入拋例外時更新流程照常完成（fail-safe
   隔離測試）。
6. **UI widget 測試**：Tab 切換、族群卡片、焦點股列表、回檔突顯與過濾、
   空狀態。
7. 全套測試綠 + 分段 code-reviewer 審查後才 commit（沿用專案慣例）。

## 風險與已接受的取捨

- **2 字白名單漏接冷門股**：接受——漏掉的本來就不是焦點股；白名單可
  隨時擴充（bump dictionaryVersion）。
- **字典演化使快照跨版本不可直接比較**：以 dictionaryVersion 欄位緩解；
  顯示層即時重算不受影響。
- **題材字典需低頻人工維護**：接受——台股題材詞彙變動速度以月計。
- **熱度標記的可能是頂部而非起點**：本功能定位為觀察清單產生器，
  進場時機交由既有三模式（回檔）判定與使用者判斷；熱度不進評分，
  未來若要進評分必須先以快照資料回測驗證。
